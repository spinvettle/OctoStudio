## 1. 项目愿景

OctoStudio 的定位已从“本地推理工具”升级为“统一 AI 网关与智能执行平台”：
对下统一管理本地与远程模型能力，对上提供 OpenAI 兼容接口、工具调用与 Agent 运行时能力。

**基础阶段能力（平台底座）**

- 模型生命周期管理：下载、校验、导入、卸载、切换。
- 推理运行时管理：`llama-server` 进程编排、健康检查、日志采集、故障恢复。
- 协议与网关能力：OpenAI 兼容 API、流式输出透传、统一错误映射。
- 路由与可用性：本地/远程统一路由、账号池、重试/熔断、超时预算。
- 可观测性：CPU/RAM/VRAM、延迟、成功率、重试与切换指标。

**上层能力（产品与智能化）**

- 交互层：多会话聊天、参数配置、聊天持久化、模型市场化 UI。
- 知识增强：RAG 检索与本地知识库问答能力。
- 工具生态：MCP Client/Server、Tool Calling 统一抽象、权限与审计。
- 智能体层：单 Agent 执行引擎、多 Agent 协作、任务编排与回放。

**核心架构：分层 Sidecar 模式**

- **主程序 (Host, Go)**
  负责网关、路由、账号池、模型管理、监控、MCP/Agent Runtime 与静态资源服务。
- **推理边车 (Sidecar, llama.cpp / 兼容推理后端)**
  负责底层推理执行与模型加载，保持计算层解耦。
- **前端 (UI, Web/Wails)**
  负责聊天、模型管理、监控面板、账号池与 Agent 工作台。

## 2. 阶段规划

### 阶段一：原型验证 (MVP) - "跑通流程"

**目标**：能够启动 Go 主程序，拉起 llama-server，并通过简易界面进行一次对话。

- [ ] **基础架构搭建**
  - 初始化 Go 项目结构 (Clean Architecture)。
  - 编写 `ProcessManager`：使用 Go `os/exec` 管理 `llama-server` 的生命周期（启动、停止、日志捕获）。
- [ ] **API 网关基础**
  - 实现反向代理：将前端请求转发给 `llama-server`。
- [ ] **手动模型加载**
  - 支持从本地指定路径加载一个 `.gguf` 模型文件启动服务。
- [ ] **最小化 UI**
  - 一个简单的输入框和输出框，验证推理链路打通。

### 阶段二：模型管理与下载系统 - "Ollama 核心能力"

**目标**：用户无需关心底层文件，通过界面即可下载并管理模型。

- [ ] **模型下载器 (Downloader)**
  - 对接 HuggingFace 或 ModelScope 镜像站 API。
  - 实现断点续传、并发下载、校验和验证 (SHA256)。
- [ ] **模型库管理 (Model Registry)**
  - 定义模型元数据结构 (manifest)，管理本地模型列表。
  - 实现模型的导入、删除、重命名。
- [ ] **资源监控 (Monitor)**
  - 集成 `gopsutil` 或 `nvidia-smi` (通过 Go 调用)。
  - 实时通过 WebSocket 向前端推送 CPU、RAM、VRAM (显存) 使用率。

### 阶段三：OpenAI 协议兼容与高级网关 - "服务化"

**目标**：让 OctoStudio 能够作为其他软件（如沉浸式翻译、VSCode 插件）的后端。

- [ ] **API 标准化**
  - 实现 `/v1/chat/completions` 接口。
  - 实现 `/v1/models` 接口。
  - 在 Go 层处理请求体格式转换（如果 `llama-server` 格式有差异），确保对外 100% 兼容 OpenAI 格式。
- [ ] **多模型并发/切换**
  - 动态卸载/加载模型：当用户请求不同模型时，自动重启 Sidecar 或动态加载权重。
- [ ] **系统托盘与后台运行**
  - 支持最小化到托盘，开机自启。

### 阶段四：账号池代理与远程容灾 - "多账户智能路由"

**目标**：在保留本地推理能力的前提下，统一代理本地与远程模型，并实现远程多账户智能切换与重试。

- [ ] **统一推理路由层 (Router)**

  - 建立 Provider 抽象：`local`、`remote-openai`、`remote-compatible`。
  - 依据策略（可用性、延迟、成本、模型能力）进行请求分发，支持本地优先或远程优先。
  - 支持流式输出透传与统一错误码映射，避免上层感知差异。

- [ ] **账号池 (Account Pool) 与密钥管理**

  - 设计 Account 实体：`provider`、`api_key`、`base_url`、`weight`、`quota`、`status`、`cooldown_until`。
  - 支持多账户分组（按模型/地区/用途）并可热更新。
  - 密钥加密存储（系统密钥链或本地加密文件），日志默认脱敏，杜绝明文泄露。

- [ ] **智能切换与重试机制**

  - 实现分层重试：同账户短重试（网络抖动）→ 跨账户切换（配额/限流）→ 可选降级模型。
  - 错误分类与熔断：针对 429/5xx/超时做指数退避、短期熔断与半开探测。
  - 负载策略可配置：轮询、加权轮询、最小延迟、成功率优先。

- [ ] **观测性与运维面板**

  - 增加路由指标：成功率、P95 延迟、每账户错误率、切换次数、重试次数。
  - 提供账号池状态页：在线/熔断/冷却中、余额或配额（若 Provider 支持）。
  - 告警策略：连续失败、全池不可用、成本超阈值。

- [ ] **兼容与安全边界**
  - 保持 OpenAI 兼容接口不变，上层客户端零改造接入。
  - 增加请求幂等键、超时预算与最大重试上限，避免雪崩和重复扣费。
  - 对高风险操作（批量切换、停用账号）增加审计日志。

### 阶段五：现代交互界面 - "CherryStudio 体验"

**目标**：提供媲美商业软件的用户体验。

- [ ] **聊天功能增强**
  - 多会话管理 (Sessions)。
  - 聊天记录持久化 (SQLite/BoltDB)。
  - 支持 System Prompt 设置、参数调节 (Temperature, Top_P)。
- [ ] **知识库 (RAG) 初步 (可选)**
  - 集成简单的向量数据库，支持本地文档问答。
- [ ] **模型市场 UI**
  - 类似于 App Store 的模型浏览、搜索、一键安装界面。

### 阶段六：生态与优化

- [ ] **跨平台打包** (Windows, macOS, Linux)。
- [ ] **GPU 自动探测** (CUDA, Metal, ROCm) 并配置最佳启动参数。
- [ ] **插件系统** (预留)。

### 阶段七：MCP 与 Agent 能力 - "从模型平台到智能执行平台"

**目标**：让 OctoStudio 从“模型推理平台”升级为“可连接工具、可执行任务、可编排流程”的智能体运行平台。

- [ ] **MCP 协议接入 (Model Context Protocol)**

  - 在 Host 层实现 MCP Client，支持连接外部 MCP Server（文件系统、数据库、浏览器自动化、企业 API）。
  - 可选实现 MCP Server，将 OctoStudio 的模型能力与路由能力反向暴露给第三方 Agent 框架。
  - 统一 Tool Schema 映射：将 MCP 工具描述转换为 OpenAI Compatible Tool Calling 结构，保证接口复用。

- [ ] **工具执行沙箱与权限系统**

  - 实现工具白名单与权限分级（只读/读写/高危），支持按会话、按用户授权。
  - 增加执行审计日志（调用者、参数摘要、执行时长、结果状态），满足可追溯要求。
  - 对高风险工具增加二次确认与超时中断机制，防止误操作扩大化。

- [ ] **单 Agent 执行引擎 (MVP)**

  - 设计 Agent Runtime：`Plan -> Act (Tool Call) -> Observe -> Reflect` 循环。
  - 支持短期记忆（会话窗口）与长期记忆（向量检索/RAG）协同。
  - 提供任务状态机（queued/running/waiting_tool/completed/failed）与可恢复执行。

- [ ] **多 Agent 协作与任务编排**

  - 增加角色化 Agent（Planner、Executor、Reviewer）并支持任务拆分与结果汇总。
  - 引入工作流 DAG 编排，支持串行/并行节点与失败重试策略。
  - 与账号池路由联动：依据任务类型自动选择本地模型或远程高能力模型。

- [ ] **观测性、成本与可靠性治理**
  - 增加 Agent 指标：任务成功率、平均步数、工具调用成功率、每任务 Token/成本。
  - 增加预算控制（单任务/单会话/单日）与熔断限流，避免失控调用。
  - 提供回放能力：重放关键推理步骤与工具调用链路，用于调试和复盘。

## 3. 可行性结论（账号池代理）

你的设想**非常可行**，而且非常适合 OctoStudio 当前方向。建议按“先可用、后智能”的顺序推进：

1. 先做统一 Router + 基础账号池；
2. 再做错误分类重试 + 熔断；
3. 最后做策略优化（按成功率/延迟/成本自适应）。

这样可以快速上线，同时把复杂度控制在可维护范围内。

## 4. 可行性结论（MCP / Agent）

在现有架构下引入 MCP 与 Agent **完全可行**，并且是自然演进方向。建议按“先连接工具、再单体智能、最后多体协作”的顺序推进：

1. 先做 MCP Client + Tool Calling 映射，让模型稳定调用外部工具；
2. 再做单 Agent Runtime（状态机 + 记忆 + 任务恢复）；
3. 最后做多 Agent 协作与工作流编排，并接入成本/可靠性治理。

这样能够在不推翻现有 OpenAI 兼容层与路由层的前提下，逐步升级为可生产化的 Agent 平台。
